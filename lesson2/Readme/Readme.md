## Урок 2. Неймспейсы и изоляция

Контейнеризация, как концепция, и технологии изоляции, используемые для реализации контейнеров, имеют глубокие корни в мире Linux и технологиях ядра операционной системы.

Идея контейнеризации начала своё развитие в Unix-подобных системах ещё в конце прошлого века, когда были введены концепции виртуальной машины (chroot) и пространства имёен пользователя (user namespaces). Они предоставляли базовую изоляцию процессов и файловой системы, позволяя создавать относительно изолированные окружения.

Однако, реализация контейнеризации в полной мере стала возможной лишь с развитием технологий ядра Linux, таких как control groups (cgroups) и namespaces (пространства имён - перевод, далее NS). Эти функции стали стандартом для создания изолированных окружений и управления ресурсами. Об этом и поговорим в этом уроке.

В контексте контейнеризации, NS представляет собой механизм операционной системы, который обеспечивает изоляцию различных системных ресурсов, путем предоставления каждому контейнеру своего собственного видимого пространства для этих ресурсов. Каждый контейнер видит только свои собственные ресурсы, такие как процессы, файловые системы, сетевые интерфейсы, и другие системные атрибуты, отделённые от других контейнеров и хоста. NS обеспечивают высокую степень изоляции, безопасности и контроля над ресурсами, создавая виртуальные среды, где контейнеры могут работать как автономные сущности, независимо друг от друга.

Когда контейнер запускается, используя технологию контейнеризации, такую как Docker, Docker внутренне использует NS в Linux для изоляции процессов, сетевых интерфейсов, файловых систем и других ресурсов. Это предотвращает возможность вмешательства контейнера в другие контейнеры и обеспечивает безопасность и изоляцию. 

Таким образом, NS позволяют контейнерам предоставить изолированные окружения, эффективно управлять ресурсами, обеспечивать безопасность и облегчать развёртывание приложений в независимых и надёжных средах.

Ниже рассмотрим типы NS с примерами их применения в Docker:

#### 1. PID Namespace (изоляция процессов)

PID namespace предоставляет изолированное пространство идентификаторов процессов для каждого контейнера. 

**Пример:**

```
docker run -d --name container1 ubuntu sleep 1000
docker run -d --name container2 ubuntu sleep 1000

```

В этом примере мы создаем два контейнера, каждый из которых выполняет команду "sleep", приостанавливающую выполнение процесса на указанное количество секунд. Поскольку каждый контейнер находится в своем собственном PID namespace, процессы внутри каждого контейнера не пересекаются с процессами в других контейнерах или на хосте.

#### 2. Mount Namespace (изоляция файловой системы)

Mount namespace создает изолированное представление файловой системы для каждого контейнера. 
**Пример:**

```
docker run -d --name container1 -v /hostdir1:/containerdir1 alpine sleep 1000
docker run -d --name container2 -v /hostdir2:/containerdir2 alpine sleep 1000

```
В данном случае каждый контейнер видит свою собственную корневую файловую систему, отделенную от файловых систем других контейнеров и хоста.

 
#### 3. Network Namespace (изоляция сетевых ресурсов)

Network namespace изолирует сетевые ресурсы, включая сетевые интерфейсы, таблицы маршрутизации, firewall правила и другие параметры сети.

**Пример:**

```
docker network create isolated_net
docker run -d --name container1 --network=isolated_net nginx
docker run -d --name container2 --network=isolated_net nginx

```
В этом примере мы создаем изолированную сеть под названием isolated_net с помощью команды docker network create. Затем мы запускаем два контейнера (container1 и container2), указывая им использовать эту изолированную сеть.

#### 4. IPC Namespace (изоляция межпроцессного взаимодействия)

IPC namespace изолирует межпроцессное взаимодействие, такое как очереди сообщений и семафоры, что позволяет контейнерам работать в изолированных средах без конфликтов взаимодействия.


#### 5. UTS Namespace (изоляция параметров системы)

UTS namespace изолирует два параметра: название хоста и доменное имя. 

**Пример:**

```

docker run --rm -it --name container1 --hostname=container1 ubuntu
docker run --rm -it --name container2 --hostname=container2 ubuntu

```

В данном примере каждый контейнер имеет своё уникальное название хоста и доменное имя, полностью отделенные друг от друга.

#### 6. User Namespace (изоляция идентификаторов пользователей)

User namespace позволяет изолировать идентификаторы пользователей и групп внутри контейнера, обеспечивая запуск процессов с привилегиями пользователя, безопасно изолированными от основной системы.

**Пример:**

```

docker run --rm -it --name container1 --user=1000:1000 ubuntu

```

Обратим внимание на эту часть: --user=1000:1000: Этот флаг указывает Docker на то, что процессы внутри контейнера должны выполняться от имени пользователя с UID (идентификатор пользователя) 1000 и GID (идентификатор группы) 1000. Это может быть полезно, если мы хотим запустить процессы в контейнере от имени определенного пользователя вместо пользователя root (UID 0)

Каждый из этих неймспейсов обеспечивает важные механизмы изоляции, позволяя контейнерам работать как независимые сущности с высокой степенью безопасности.

Далее рассмотрим контрольные группы (cgroups) - это механизм ядра Linux, предоставляющий функции для ограничения, приоритезации и изоляции ресурсов (таких как CPU, память, ввод/вывод и другие) для группы процессов. В контексте контейнеризации, cgroups играют важную роль в управлении и контроле вычислительных ресурсов, предоставляемых контейнерам.

#### Роль cgroups в контейнеризации

1. Управление ресурсами: Cgroups позволяют ограничивать доступ контейнеров к ресурсам, таким как ЦП, память и ввод/вывод, обеспечивая равномерное распределение ресурсов между контейнерами.
   
2. Изоляция: Cgroups обеспечивают изоляцию ресурсов между контейнерами, что позволяет контейнерам работать независимо друг от друга, не конкурируя за ресурсы.
   
3. Приоритезация: Путем назначения приоритетов ресурсам, контрольные группы позволяют выделять важные ресурсы для определенных контейнеров, обеспечивая их поддержание в приоритете.

Давай рассмотрим пример использования cgroups в контексте контейнеризации с помощью Docker:

1. Ограничение ресурсов: Предположим, у нас есть несколько контейнеров, и мы хотим установить лимит на использование ЦП и памяти для каждого контейнера. Мы можем использовать cgroups для установки этих лимитов, обеспечивая справедливое распределение ресурсов между контейнерами.

2. Изоляция ресурсов: Cgroups позволяют каждому контейнеру иметь свою долю ресурсов, взаимодействуя непосредственно с ядром операционной системы для контроля и управления этими ресурсами. Это предотвращает возможность воздействия одного контейнера на ресурсы другого контейнера.

3. Приоритезация: Cgroups также позволяют устанавливать приоритеты доступа к ресурсам, что позволяет выделить важные ресурсы определенным контейнерам, обеспечивая их стабильную и предсказуемую работу.

Кратко скажем о других технологиях, которые обеспечивают изоляцию в контейнеризации:

1. Seccomp (Secure Computing Mode): Seccomp позволяет ограничивать доступ процессов к системным вызовам ядра Linux. В контексте контейнеризации, Seccomp может быть использован для блокировки определенных системных вызовов, что уменьшает поверхность атаки и повышает безопасность контейнеров. Например, контейнер может использовать Seccomp для разрешения только определенных системных вызовов, таких как read, write, exit, и блокировки опасных вызовов, таких как socket, setuid и др.

   Пример использования Seccomp в Docker:
   
   docker run --security-opt seccomp=unconfined my_container
   
   В этом примере опция seccomp=unconfined отключает ограничения Seccomp для контейнера.

2. AppArmor: В контексте контейнеризации, AppArmor позволяет определять профили безопасности для контейнеров, ограничивая их доступ к файлам, сетевым ресурсам и другим объектам в операционной системе хоста. Например, можно создать профиль безопасности для контейнера, который разрешает доступ только к определенным каталогам и файлам, предотвращая доступ к другим частям файловой системы хоста.

   Пример использования AppArmor в Docker:
   
   docker run --security-opt apparmor=my_profile my_container
   
   В этом примере опция apparmor=my_profile указывает Docker использовать профиль безопасности my_profile для контейнера.

3. SELinux (Security-Enhanced Linux): В контексте контейнеризации, SELinux обеспечивает дополнительный уровень контроля над доступом к ресурсам и объектам в системе. Например, можно определить политику SELinux, которая разрешает контейнерам доступ только к определенным файлам и директориям в файловой системе хоста.

1. Изолированное пространство идентификаторов процессов для каждого контейнера.

4. Изоляцию межпроцессного взаимодействия, такое как очереди сообщений и семафоры.

Правильный ответ: 1. Изолированное пространство идентификаторов процессов для каждого контейнера.

Объяснение: PID Namespace обеспечивает изоляцию процессов, разделяя идентификаторы процессов между контейнерами, таким образом, каждый контейнер видит только свои процессы, не имея доступа к процессам других контейнеров или хоста.
